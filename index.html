  • Thread Lifecycle Explained

    1. Introduction

    Understanding thread lifecycle is crucial for building robust, high-performance concurrent applications. We'll explore both OS-level and JVM perspectives.

    2. OS-Level Thread Lifecycle

    • New (Created)
      • Thread control block (TCB) allocated
      • Resources initialized (stack, registers)
    • Ready
      • On run queue waiting for CPU
      • Scheduler selects based on priority
    • Running
      • CPU executing instructions
      • Context switches handle state transitions
    • Blocked (Waiting)
      • Waiting for I/O, lock, or event
      • Not eligible for CPU
    • Terminated (Zombie)
      • Execution finished/killed
      • Resources freed after status collection

    3. Java Thread States (java.lang.Thread.State)

    State Description Entry Trigger Exit Trigger
    NEW Thread created but not started new Thread() start()
    RUNNABLE Eligible to run start()/notify() sleep()/wait()
    BLOCKED Waiting for monitor lock synchronized contention Lock acquired
    WAITING Indefinite wait for action wait()/join() notify()/interrupt()
    TIMED_WAITING Waiting with timeout sleep(ms)/wait(ms) Timeout/notification
    TERMINATED Execution completed run() exits -

    4. Key Lifecycle Transitions

    • Creation: new Thread() → start()
    • Running → Blocking:
      • sleep() → TIMED_WAITING
      • wait() → WAITING
      • Lock contention → BLOCKED
    • Blocking → Runnable:
      • Lock release → RUNNABLE
      • notify() → BLOCKED → RUNNABLE
    • Termination: run() completes → TERMINATED

    5. Memory Consistency (Happens-Before)

    • Thread start: Writes before start() visible in new thread
    • Lock release/acquire: Unlock happens-before subsequent lock
    • Volatile variables: Write happens-before subsequent read
    • Thread join: Termination happens-before join() returns

    6. Practical Q&A

    • BLOCKED vs WAITING: BLOCKED waits for lock, WAITING for notification
    • sleep() vs wait(): sleep() keeps locks, wait() releases monitor
    • Daemon threads: thread.setDaemon(true) before start()
    • Interrupting: thread.interrupt() throws InterruptedException

    7. Interview Tips

    • Draw state diagrams with transition triggers
    • Distinguish kernel vs user-level threads
    • Discuss thread pools (Executors.newFixedThreadPool)
    • Explain context switching costs

  • The Executor framework decouples task submission from execution mechanics. Executors provide thread pools (FixedThreadPool, CachedThreadPool) and scheduling (ScheduledThreadPool).

  • CompletableFuture in Java 8+ supports building async pipelines: thenApplyAsync, thenCombine, exceptionally, allOf/anyOf, etc.

  • Use synchronized blocks, locks (ReentrantLock), concurrent collections, or atomic classes (AtomicInteger, AtomicReference) to coordinate access.

  • Java Strings are immutable and stored in the String Pool to maximize reuse. From Java 7 on, the pool lives on the heap.

  • StringBuffer is synchronized (thread-safe) but slower. StringBuilder is unsynchronized and faster for single-thread use.

  • You can use new StringBuilder(str).reverse().toString() or write a loop swapping characters in a char array.
